# Spring Data JPA & Hibernate — Entity Lifecycle, Fetch, N+1, JPQL, Caching

📁 Package: `com.catmanscode.spring.jpa`

---

## JPA vs Hibernate

JPA is the specification (Jakarta Persistence 3.x); Hibernate is the most common implementation. Spring Data JPA wraps Hibernate via the `EntityManager` SPI, adding repository abstractions.

`EntityManagerFactory` (thread-safe, expensive to create — one per persistence unit) → `EntityManager` (not thread-safe, one per transaction) → `Session` (Hibernate-specific extension of `EntityManager`).

Spring manages `EntityManager` via a proxy: `@PersistenceContext EntityManager em` injects a thread-bound proxy that delegates to the current transactional `EntityManager`. Calling `em` outside a transaction creates a temporary session per call (resource-local).

## Entity Lifecycle

```
New (Transient)
  → persist()     → Managed
                       ↓ flush/commit → Synchronized with DB
  → detach()      → Detached
  → merge(detached) → Managed (new proxy, re-attached)
  → remove()      → Removed → Deleted on flush
```

**Dirty checking:** Hibernate snapshots the state of all managed entities at load time. On flush (`COMMIT`, explicit `flush()`, or before a query in the same session by default), it diffs each entity against its snapshot and issues `UPDATE` only for changed fields. This is why you do not call `save()` inside a transaction to update — mutation alone is enough.

**Flush modes:**
- `AUTO` (default): flushes before queries that might be affected by pending changes
- `COMMIT`: flushes only on commit — risky if in-session queries need current state
- `MANUAL`: explicit flush required — used with `@Transactional(readOnly = true)` to avoid dirty checking

## Fetch Strategies

| Strategy | When loaded | Risk |
|----------|-------------|------|
| `EAGER` | Always, with parent | Over-fetching; JOIN or N+1 |
| `LAZY` | On first access | `LazyInitializationException` outside session |

JPA defaults: `@ManyToOne` and `@OneToOne` default to `EAGER`; `@OneToMany` and `@ManyToMany` default to `LAZY`. **Override both to `LAZY`** — control fetching at the query level, not the mapping level.

### `LazyInitializationException`

Occurs when a lazy association is accessed outside an open `EntityManager` (after the transaction commits). Common in:
- REST controller accessing `entity.getChildren()` after the service method returns
- `toString()` / Jackson serialization triggering lazy load

**Fixes:**
- Fetch joins in JPQL: `JOIN FETCH u.orders`
- `@EntityGraph` — declarative fetch plan without modifying JPQL
- DTO projection — select only needed columns, no entity at all
- **Open Session in View (OSIV):** keeps session open through the view layer — defaults to `true` in Spring Boot. Convenient but dangerous: lazy loads fire during serialization without transaction, can cause N+1, and leaks DB connections. Disable in production: `spring.jpa.open-in-view=false`.

## N+1 Query Problem

Symptom: loading N parent entities triggers N additional queries to fetch their association, one per parent.

```java
// 1 query for users + N queries for each user's orders
List<User> users = userRepo.findAll();
users.forEach(u -> u.getOrders().size());
```

**Detection:** Hibernate statistics (`hibernate.generate_statistics=true`), `p6spy`, `datasource-proxy`.

**Fixes:**

1. **JOIN FETCH** in JPQL:
   ```java
   @Query("SELECT u FROM User u JOIN FETCH u.orders")
   List<User> findAllWithOrders();
   ```
   Problem: returns duplicates (SQL JOIN semantics) — add `DISTINCT` or use `Set`.

2. **`@EntityGraph`:**
   ```java
   @EntityGraph(attributePaths = {"orders"})
   List<User> findAll();
   ```

3. **`@BatchSize`:** Hibernate fetches lazy associations in batches of N (`IN` clause) instead of one-by-one. Reduces N+1 to `N/batchSize + 1` queries. A global default: `hibernate.default_batch_fetch_size=25`.

4. **Subselect fetching:** `@Fetch(FetchMode.SUBSELECT)` — fetches all associations in a single subselect after the root query. Good for large result sets.

5. **DTO projections** — the cleanest solution for read-only endpoints:
   ```java
   @Query("SELECT new com.example.dto.UserSummary(u.id, u.name, COUNT(o)) FROM User u LEFT JOIN u.orders o GROUP BY u.id, u.name")
   List<UserSummary> findUserSummaries();
   ```

## JPQL and Criteria API

**JPQL** operates on entity model (classes and fields), not tables and columns. `@NamedQuery` is validated at startup — fail-fast for typos. `@Query` in Spring Data is validated at startup only if the provider supports it.

**Criteria API** — programmatic, type-safe query building. Verbose but essential for dynamic filters. Metamodel (`User_`) is generated by annotation processor.

**Native queries** (`@Query(nativeQuery = true)`) — bypass JPA, use when JPQL cannot express the query (window functions, CTEs, database-specific functions). Results are `Object[]` or interface-based projections.

**Projections:**
- Interface-based: Spring Data generates a proxy, each getter maps to an alias in the query
- Class-based (DTO): constructor expression in JPQL or `@Value` on interface method
- `Tuple` — JPA standard; similar to `Object[]` but accessed by alias

## Caching

### First-Level Cache (Session Cache)
Scoped to the `EntityManager`/`Session`. Guarantees identity within a transaction — `em.find(User.class, 1L)` twice returns the same instance. Cannot be disabled; cleared by `em.clear()`.

### Second-Level Cache (L2C)
Shared across sessions, scoped to `EntityManagerFactory`. Requires an explicit provider (Ehcache, Caffeine, Hazelcast). Configure per-entity:

```java
@Entity
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Product { ... }
```

| Strategy | Use when |
|----------|---------|
| `READ_ONLY` | Immutable data (reference tables) |
| `NONSTRICT_READ_WRITE` | Rarely updated; stale reads acceptable |
| `READ_WRITE` | Frequently updated; soft-lock on write |
| `TRANSACTIONAL` | JTA environments; full transactional cache |

### Query Cache
Caches query result *identifiers* (not entities — those come from L2C). Only useful when the same parameterized query runs repeatedly and the result set rarely changes. Invalidated by any write to the queried tables — can thrash under write load.

## Common Interview Pitfalls

- `save()` on a managed entity inside a transaction is a no-op — Hibernate dirty-checks automatically. Calling `save()` is not wrong but indicates a misunderstanding of entity state.
- `merge()` does NOT update the passed detached entity — it returns a NEW managed entity. The detached instance is still detached after the call.
- `CascadeType.REMOVE` on `@OneToMany` deletes children one by one (Hibernate loads them to fire lifecycle events). For bulk deletes, use `@Query("DELETE FROM Child c WHERE c.parent.id = :pid")`.
- `orphanRemoval = true` on `@OneToMany` removes a child from DB when it is removed from the parent's collection — even without explicit `remove()`. Combining with `CascadeType.PERSIST` is the standard bidirectional ownership pattern.
- Enabling L2C without a proper eviction strategy in a clustered environment leads to stale reads — each node has its own cache unless a distributed provider (Hazelcast, Infinispan) is used.
